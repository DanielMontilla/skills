# When To Use

When user asks to spec a new feature, break a feature into tasks, create a feature plan, or when description/requirements have gaps that need resolution before proceeding.

# Pipeline

## 1. Gather Input

Ask user for feature name (kebab-case), description, requirements, goals. If gaps exist in requirements, load [grill-me](../grill-me/SKILL.md) skill to resolve. Resolve all gaps before proceeding.

- Feature dir: `.agents/features/<kebab-name>/`
- Task dir: `<LETTER><NN>-<kebab-task-name>/`
- Defect dir: `<task-dir>/defects/<LETTER><NN>-<kebab-desc>/`

## 2. Analyze Dependencies

Identify foundation layer (A) — tasks others depend on. Identify parallel layers (B, C, ...) — tasks dependent on previous layer. Assign letter prefix per group, number per task (01-99).

Group rules:
- Same letter prefix = same group = parallel execution
- Default: Group N depends on all M where letter < N
- Per-task `depends-on` in TASK.md frontmatter overrides group-level rule and is system of record for eligibility
- Prefer larger groups → more parallelism

Example: A01 → B01, B02 → C01, C02, C03

## 3. Design Gates

GATES.md files gate verification of feature scope, tasks, and optionally defects. Each GATES.md is optional; feature root and most tasks have one, low-complexity tasks may skip.

Where GATES.md can live:
- Feature root (`<feature-dir>/GATES.md`) — feature-level gates
- Task dir (`<task-dir>/GATES.md`) — task-level gates (most tasks)
- Defect dir (`<defect-dir>/GATES.md`) — only for defects NOT generated by a gate (i.e. `generated: orchestrator` or `generated: user`)

Gate structure:
- Gates grouped into phases 0..N
- Phase 0 is entry point (no prerequisite)
- Phase N requires Phase N−1 fully passed before starting
- All gates within a same phase execute in parallel by sub-agents with no shared memory/inter-agent communication
- The sub-agent validating a gate writes the gate's own status back into GATES.md
- Per-gate status values: `pending | in-progress | passed | failed`

Typical gates (orchestrator chooses per scope):
- Formatting check (`bun run format:check` or equivalent)
- Lint check
- Test check
- TypeScript conventions check (load `typescript-conventions` skill)
- Final phase: very strict final conventions check (comprehensive conventions review)

When a gate fails, it may generate a defect at `<task-dir>/defects/<LETTER><NN>-<kebab-desc>/` with frontmatter `generated: gate`. Defects with `generated: gate` do NOT get their own GATES.md — the generating gate is their gate. The parent GATES.md tracks all generated defects in a `## Defects History` section.

## 4. Present Plan

Show directory tree, dependency graph (A→B→C), gate phases per scope. Preview file templates from `templates/`. Ask user confirmation. Wait for explicit approval before generating.

## 5. Generate Files

Create `.agents/features/<feature-name>/` dir and all subdirectories. Generate files from templates:
- `FEATURE.md` — feature overview with tasks, dependencies, gates table, progress
- `CHECKLIST.md` — feature progress tracker with status per task group
- `TASK.md` + `MEMORY.md` per task directory
- `GATES.md` per scope that needs gates (feature root, task dirs, eligible defect dirs)

Apply caveman-compression to all files. Plans are living documents — executors may update TASK.md when they discover a better approach; changes documented in MEMORY.md → Deviations.

Conventions:
- All files use caveman-compression
- Frontmatter allowed for structured data
- Task names short, descriptive, unique within feature
- Status values: `pending | in-progress | complete | defect`
- Defects with `generated: gate` NEVER have a GATES.md file
